# 不正ログイン監査基盤 - 手動デプロイガイド

## 前提条件

- AWS CLI インストール済み
- 3つのCloudFormationテンプレートをデプロイ済み
- EC2 キーペア作成済み
- VPC/サブネット作成済み

---

## Step 1: CloudFormation スタックデプロイ

### 1.1 CloudTrail 基盤層

```bash
aws cloudformation create-stack \
  --stack-name login-audit-cloudtrail \
  --template-body file://01-cloudtrail-foundation.yaml \
  --capabilities CAPABILITY_IAM \
  --region ap-northeast-1

# 完了確認
aws cloudformation wait stack-create-complete \
  --stack-name login-audit-cloudtrail \
  --region ap-northeast-1

# 出力確認
aws cloudformation describe-stacks \
  --stack-name login-audit-cloudtrail \
  --query 'Stacks[0].Outputs[*].[OutputKey,OutputValue]' \
  --region ap-northeast-1
```

**記録すること**:
- `CloudTrailLogsBucket` (例: `login-audit-ctlogs-123456789012`)


## Step 2: Redshift 初期化

### 2.1 接続確認

```bash
# Redshift エンドポイント取得
REDSHIFT_ENDPOINT=$(aws cloudformation describe-stacks \
  --stack-name login-audit-ec2-redshift \
  --query 'Stacks[0].Outputs[?OutputKey==`RedshiftEndpoint`].OutputValue' \
  --output text \
  --region ap-northeast-1)

echo $REDSHIFT_ENDPOINT
```

### 2.2 SQL ファイルアップロード

```bash
# EC2 に SSH 接続 (Bastion 経由または Direct Connect)
# ローカルから EC2 へ redshift_init.sql をコピー
scp -i your-key.pem redshift_init.sql ec2-user@<EC2_PRIVATE_IP>:/tmp/

# または SSM 経由でアップロード
aws ssm send-command \
  --instance-ids i-0xxxxxxxxxxxx \
  --document-name "AWS-RunShellScript" \
  --parameters 'commands=["mkdir -p /opt/audit-etl/scripts"]' \
  --region ap-northeast-1

aws s3 cp redshift_init.sql s3://your-temp-bucket/

aws ssm send-command \
  --instance-ids i-0xxxxxxxxxxxx \
  --document-name "AWS-RunShellScript" \
  --parameters 'commands=["aws s3 cp s3://your-temp-bucket/redshift_init.sql /opt/audit-etl/scripts/"]' \
  --region ap-northeast-1
```

### 2.3 SQL 実行

```bash
# EC2 上で実行
ssh -i your-key.pem ec2-user@<EC2_PRIVATE_IP>

# Redshift 接続テスト
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 -c "SELECT version();"

# 初期化 SQL 実行
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -f /opt/audit-etl/scripts/redshift_init.sql

# 確認
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -c "\dt audit.*"
```

**確認項目**:
- `audit.console_login_stage`
- `audit.console_login_audit`
- `audit.daily_login_summary`
- `audit.anomaly_history`
- `audit.etl_job_history`
- `audit.user_login_stats`
- `audit.v_high_risk_logins` (VIEW)
- `audit.v_daily_summary` (VIEW)
- `audit.v_unacknowledged_anomalies` (VIEW)

## Step 3: ETL スクリプトデプロイ

### 3.1 スクリプトアップロード

```bash
# 方法1: SCP (EC2 に直接アクセス可能な場合)
scp -i your-key.pem audit_etl.py ec2-user@<EC2_PRIVATE_IP>:/opt/audit-etl/scripts/

# 方法2: S3 経由 + SSM
aws s3 cp audit_etl.py s3://your-temp-bucket/

aws ssm send-command \
  --instance-ids i-0xxxxxxxxxxxx \
  --document-name "AWS-RunShellScript" \
  --parameters 'commands=[
    "aws s3 cp s3://your-temp-bucket/audit_etl.py /opt/audit-etl/scripts/",
    "chmod 644 /opt/audit-etl/scripts/audit_etl.py",
    "chown ec2-user:ec2-user /opt/audit-etl/scripts/audit_etl.py"
  ]' \
  --region ap-northeast-1
```

### 3.2 環境変数ファイル設定

```bash
# EC2 に SSH 接続
ssh -i your-key.pem ec2-user@<EC2_PRIVATE_IP>

# 環境変数ファイル編集
sudo vi /opt/audit-etl/config/env.conf
```

**env.conf 内容**:
```bash
export PROJECT_NAME=login-audit
export ENVIRONMENT=audit
export ATHENA_WORKGROUP=login-audit-audit-workgroup
export ATHENA_DATABASE=login-audit_audit_db
export S3_STAGE_BUCKET=your-stage-bucket
export CLOUDTRAIL_BUCKET=login-audit-ctlogs-123456789012
export REDSHIFT_ENDPOINT=login-audit-audit-cluster.xxx.ap-northeast-1.redshift.amazonaws.com
export REDSHIFT_DATABASE=audit
export REDSHIFT_USERNAME=audit_admin
export REDSHIFT_PASSWORD=YourActualPassword123
export AWS_REGION=ap-northeast-1

# 深夜判定時間 (24時間制)
export NIGHTTIME_START=22
export NIGHTTIME_END=6
```

```bash
# 権限設定
sudo chmod 600 /opt/audit-etl/config/env.conf
sudo chown ec2-user:ec2-user /opt/audit-etl/config/env.conf
```

### 3.3 cron 有効化

```bash
# cron 設定ファイルを有効化
sudo mv /etc/cron.d/audit-etl.disabled /etc/cron.d/audit-etl

# cron サービス再起動
sudo systemctl restart crond

# 確認
cat /etc/cron.d/audit-etl
crontab -l -u ec2-user
```

## Step 4: 動作確認

### 4.1 手動実行テスト

```bash
# EC2 上で手動実行
sudo su - ec2-user
source /opt/audit-etl/config/env.conf
python3 /opt/audit-etl/scripts/audit_etl.py
```

### 4.2 ログ確認

```bash
# ETL 実行ログ
tail -f /var/log/audit-etl/ETL-*.log

# cron ログ
tail -f /var/log/audit-etl/cron.log
```

### 4.3 Redshift データ確認

```bash
# 本番テーブル確認
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -c "SELECT COUNT(*) FROM audit.console_login_audit;"

# 日次集計確認
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -c "SELECT * FROM audit.v_daily_summary ORDER BY audit_date DESC LIMIT 5;"

# 高リスクログイン確認
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -c "SELECT * FROM audit.v_high_risk_logins ORDER BY risk_score DESC LIMIT 5;"

# 未対応異常確認
psql -h $REDSHIFT_ENDPOINT -U audit_admin -d audit -p 5439 \
  -c "SELECT * FROM audit.v_unacknowledged_anomalies;"
```

### 4.4 Athena クエリ確認

```bash
# Athena クエリ実行テスト
aws athena start-query-execution \
  --query-string "SELECT COUNT(*) FROM cloudtrail_console_login WHERE year='2026' AND month='07' AND day='20'" \
  --work-group login-audit-audit-workgroup \
  --query-execution-context Database=login-audit_audit_db \
  --region ap-northeast-1
```

## トラブルシューティング

### ETL 失敗時

```bash
# 最新ログ確認
cd /var/log/audit-etl
ls -lt | head -5
tail -100 ETL-*.log

# Athena クエリ履歴確認
aws athena list-query-executions \
  --work-group login-audit-audit-workgroup \
  --region ap-northeast-1
```

### Redshift 接続失敗

```bash
# セキュリティグループ確認
aws ec2 describe-security-groups \
  --group-ids sg-xxxxxxxx \
  --query 'SecurityGroups[0].IpPermissions'

# Redshift クラスター状態確認
aws redshift describe-clusters \
  --cluster-identifier login-audit-audit-cluster \
  --query 'Clusters[0].ClusterStatus' \
  --region ap-northeast-1
```

### cron 実行されない

```bash
# cron サービス状態確認
sudo systemctl status crond

# cron ログ確認
cat /var/log/cron | grep audit-etl
cat /var/log/audit-etl/cron.log
```
